## 4. 解説が短い問題の扱い方

### 4-1. 基準設定

| 分類 | 基準 | 対応方針 |
|---|---|---|
| **要拡充** | 解説が1文（約60文字未満）かつ他の選択肢の誤り理由が不明 | Gemini文案作成→THINK判断→Codex実装 |
| **推奨拡充** | 解説が2文（約100文字未満）かつ法改正説明が欠落 | 同上（優先度：中） |
| **現状維持** | 解説が3文以上または論点が単純で短くても十分な場合 | 対応不要 |

### 4-2. 現在の要拡充問題

| ID | 現在の解説文字数 | 拡充内容 |
|---|---|---|
| TK-2026-0001 | 約57文字（1文） | 他の選択肢（法人破産と役員個人の関係・例外なし規定の誤り）の説明を追記 |
| TK-2026-0038 | 約102文字（2文）※改正説明なし | 2020年民法改正前後の危険負担ルール変化を追記 |

### 4-3. 解説拡充の基本方針

- 解説の目的：「なぜ正解が正解か」だけでなく「なぜ他の選択肢が誤りか」を説明する
- 目安文字数：100〜200文字（3〜4文）
- 法改正・制度変更の注記は「（※注意）」で留めず、改正の具体的内容を記述する

## 5. verified=true 移行方針

### 5-1. 現状

| 項目 | 数値 |
|---|---|
| 全問題数 | 300問 |
| verified=true | 2問（TK-2026-0079・TK-2026-0099） |
| verified=false | 292問（新形式） |
| verifiedフィールドなし | 8問（旧形式ID整数型） |

### 5-2. 推奨方針

**「レビュー完了した問題から順にverified=trueへ移行する漸進的アプローチ」を採用する。**

#### 移行条件（1問あたり）
1. 問題文・選択肢・解説の内容を人間が読んで確認した
2. 法的正確性に問題がない（または修正済みである）
3. 正解（correct値）が正しい

#### 移行優先順序
1. Phase 7-Dで修正済みの3問（TK-2026-0023 / TK-2026-0082 / TK-2026-0095）→ 次のPhase 7-Fで修正後に付与
2. Phase 7-F で修正・確認する問題（TK-2026-0038 / TK-2026-0145 / TK-2026-0083 / TK-2026-0001）→ 修正完了後に付与
3. 類似問題整理で「残す」と決定した問題→ THINKレビュー後に付与

#### 移行しない条件
- verified=false のまま大量に追加問題を作成する段階（500問化フェーズ）は、追加問題を先に作りverified移行は500問完成後に一括で行う

### 5-3. confidence フィールドの扱い

| 値 | 意味 | 移行タイミング |
|---|---|---|
| "unchecked" | AI生成のまま未確認 | verified=falseのまま |
| "medium" | 部分的に確認済み | 任意でverified=true付与可 |
| "high" | 法律確認・人間レビュー済み | verified=true付与を推奨 |

## 9. THINK が最終判断すべき項目

以下はGemini確認・Codex集計の後でTHINKが意思決定する必要がある。

### 9-1. 削除する問題の確定（完全類似整理）

完全類似グループ約15件について、**どちらの問題を残すか・削除するか**をTHINKが確定する。

> 考え方：削除する問題を選ぶ基準
> 1. 解説の情報量が少ない方を削除
> 2. 論点が狭い（1点のみ）方を削除
> 3. verified=falseかつ修正が必要な方を削除（修正コストを避ける）
> 4. 問題番号が後ろの方を削除（先の問題が先に作られた可能性が高い）

### 9-2. verified=true 付与タイミング

Phase 7-D修正済み3問（TK-0023・TK-0082・TK-0095）に今すぐverified=trueを付与するか、それとも全整理完了後に一括付与するかを判断する。

> 推奨：修正済み問題から順次付与（漸進的アプローチ）

### 9-3. 解説拡充のスコープ

「法的正確性の修正（必須）」と「教材品質向上（任意）」のどちらまで Phase 7-F に含めるかを決定する。

> 推奨：Phase 7-Fは必須（A分類6問の修正）に集中。教材品質向上（B分類）は並行または次フェーズに回す。

### 9-4. クーリング・オフの4問整理（TK-0023 / TK-0106 / TK-0145 / TK-0246）

この4問のうち残す問題（2〜3問）を指定する。各問題の論点：
- TK-0023：自宅指定申込みは除外（論点：除外場所）
- TK-0106：書面発信主義（論点：通知方法・効力発生時点）
- TK-0145：8日間の起算点（論点：期間計算）
- TK-0246：自宅指定申込みは除外（論点：除外場所）→ 0023と重複

### 9-5. correct=1偏り（30.3%）への対応要否

追加問題作成時の指針として「correct=0またはcorrect=3に偏らせる」指針を設けるかどうかを判断する。

> 推奨：指針設定不要。500問化時に新規問題で自然分散させれば十分。

### 9-6. 問題削除後の穴埋め方針

完全類似問題の削除によって問題数が均等（各75問）から崩れた場合、500問化フェーズ前に補完するかどうかを判断する。

> 推奨：削除問題数が各カテゴリ5問以内であれば500問化フェーズでまとめて追加する。

## 10. Phase 7-F の推奨実施順序

```
Step 1: Codex集計（解説文字数・注記付き問題・類似候補の全文比較表）
        → 出力：集計レポート / 類似問題全文比較表

Step 2: Gemini確認（TK-0038・TK-0083・TK-0145・TK-0001 の法的正確性・解説文案）
        → 出力：修正案レポート

Step 3: THINK判断①（類似問題の削除問題確定）
        → 出力：削除リスト（問題IDの一覧）

Step 4: Codex実装①（類似問題の削除）
        → 変更ファイル：questions-*.js
        → verify → commit

Step 5: THINK判断②（A分類問題の修正案承認）
        → 出力：承認または修正指示

Step 6: Codex実装②（A分類問題の解説修正・解説拡充）
        → 変更ファイル：questions-*.js
        → verify → commit

Step 7: verified=true 付与（修正・確認済み問題）
        → 変更ファイル：questions-*.js
        → verify → commit

Step 8: 500問化前の品質基準チェック
        → verify クリア・カテゴリバランス確認
        → Phase 7-F 完了宣言
```
