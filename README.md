## ブラウザで開く方法

`index.html` をダブルクリックするか、ChromeやSafariにドラッグ＆ドロップするだけで動作します。

スマホのSafari/Chromeからファイルを開いてください。

## 問題を追加する方法

問題は**分野別ファイル**に追加します。`questions.js` は統合専用ファイルのため、**直接編集しないでください**。

| 分野 | ファイル |
|---|---|
| 宅建業法 | `questions-takken.js` |
| 権利関係 | `questions-rights.js` |
| 法令上の制限 | `questions-laws.js` |
| 税・その他 | `questions-tax.js` |

追加する問題の形式：

```js
{
  id: "TK-2026-0101",          // ← TK-YYYY-NNNN 形式（他と重複しない文字列）
  category: "宅建業法",          // ← 上記4つのうちどれか
  question: '問題文をここに書く',
  options: [
    '選択肢1',
    '選択肢2',
    '選択肢3',
    '選択肢4'
  ],
  correct: 0,                   // ← 正解のインデックス（整数 0〜3）
  explanation: '解説文をここに書く',
  verified: false,              // ← AI生成・未確認の場合は false（新形式IDでは必須）
},
```

**使えるカテゴリ（全4種）**
- `"宅建業法"`
- `"権利関係"`
- `"法令上の制限"`
- `"税・その他"`

**追加後は必ず以下を実行し、エラー 0 件を確認してください。**

```powershell
powershell -ExecutionPolicy Bypass -File .\verify-questions.ps1
```

## Codexにレビューさせる際の観点

以下の点を重点的に確認してもらうと良いです。

**1. 問題の正確性（questions-*.js）**
- 各選択肢の内容が現行法令と一致しているか
- `correct` インデックスと解説の正答が一致しているか
- 2024年以降の法改正が反映されているか

**2. ロジックの堅牢性（script.js）**
- `startQuiz` でquestions配列が空の場合のハンドリング
- `answerQuestion` の二重送信防止（`selectedAnswer !== null` ガード）
- `localStorage` 読み書きの try/catch

**3. 拡張性（将来への備え）**
- 問題IDの採番ルール（新形式: TK-YYYY-NNNN 文字列）
- カテゴリの追加方法（`CATEGORIES` 配列と `icons` オブジェクトの2か所を更新）
- 1問ずつ正誤を記録している現構造で「直近X回の正答率」が取れるか

**4. スマホUX**
- タップ領域が44px以上確保できているか（iOS HIG基準）
- 長い問題文が画面内に収まるか（端末幅375px基準）
- SafariのSafe Area（ノッチ・ホームインジケーター）対応

**5. アクセシビリティ**
- 色だけに頼らない正誤表示になっているか（アイコン併用）
- disabled状態のボタンのコントラスト比

## 問題データ運用ルール

問題を追加・編集するときは以下のルールを守ってください。

### IDの管理
- **問題IDは一度使ったら変更・再利用しない。**
  `localStorage` の学習履歴はIDで紐づいているため、IDを変えると過去データと不整合が起きます。
  削除した問題のIDは欠番のまま残し、次の問題は続きの番号を使ってください。

### 正解位置のバランス
- **問題を追加するときは `correct: 0/1/2/3` が偏らないようにする。**
  直前10問の正解分布を確認し、同じ番号が続いているときは別の位置に正解を設定してください。

### AI生成問題の扱い
- **AI（ChatGPT・Claude・Gemini等）で生成した問題は `verified: false` を必ず設定する。**
  公式テキストや過去問で内容を確認したら `verified: true` に変更し、`source` フィールドに出典を記入してください。
  `verified` フィールドのない新形式ID問題はエラーになります（`verify-questions.ps1` で検出）。

### 現在使用中のオプションフィールド
新形式ID（TK-YYYY-NNNN）の問題では以下のフィールドが使用されています。

```js
{
  id: "TK-2026-0001",
  // ...必須フィールド...
  verified: false,              // 公式確認済みかどうか（新形式IDでは必須・boolean）
  source: 'AI-generated draft', // 出典
  difficulty: 'basic',          // 'basic' | 'standard' | 'hard'
  confidence: 'unchecked',      // 確認状態
  generatedBy: 'Gemini',        // 生成AI名
  lawUpdatedAt: '2026-05-24',   // 参照した法令の最終改正年月
}
```

## 問題データ検証スクリプト（verify-questions.ps1）

### 目的

`questions-takken.js` / `questions-rights.js` / `questions-laws.js` / `questions-tax.js`
の問題データを自動的にチェックし、構造上の問題を検出・報告します。
**問題データの自動修正は行いません。検出と報告のみです。**
Node.js 不要、PowerShell 単体で動作します。

---

### 実行方法

プロジェクトのルートフォルダで以下のコマンドを実行します。

```powershell
powershell -ExecutionPolicy Bypass -File .\verify-questions.ps1
```

エラーがなければ終了コード `0`、1件でもエラーがあれば終了コード `1` で終了します。

---

### 検証項目（Level 1）

| # | 内容 |
|---|---|
| 1 | 対象ファイルが存在するか |
| 2 | ファイルごとの `id` 数をカウント |
| 3 | 全ファイル横断で `id` の重複チェック |
| 4 | `question` の存在数をカウント・空文字チェック |
| 5 | `options` の存在数をカウント・選択肢数(4個)/空文字/重複チェック |
| 6 | `correct` の存在数をカウント・値範囲チェック（整数 0〜3） |
| 7 | `explanation` の存在数をカウント・空文字チェック |
| 8 | `verified` の存在数カウント・boolean 値チェック（新形式ID必須） |
| 9 | カテゴリ名の出現数を集計 |
| 10 | 総問題数を表示 |
| 11 | 新形式ID の形式チェック（`^TK-\d{4}-\d{4}$`） |

**整合性チェック：** `id` 数を基準に、各フィールドの数が一致しているかを確認します。
一致しない場合はエラーとして報告します。

**IDの扱い：** `TK-2026-0001` のような文字列IDも、`3` のような整数IDも、
すべて**文字列として**比較します（`parseInt` は行いません）。

**`verified` の扱い：**
- `TK-形式ID`（新形式）の問題には `verified` が必須 → なければエラー
- `verified` の値は `true` または `false`（boolean）であること → 文字列 `"false"` はエラー
- 旧形式ID（整数）の問題には `verified` がなくても仕様通り → エラーなし

---

### エラーが出た場合の対応方針

1. **エラー内容をそのまま修正しない** — まず何が問題かを確認する
2. **データの意図を確認する** — タイポか、仕様変更か、テキストの問題かを切り分ける
3. **問題データのバックアップを取ってから修正する** — `git diff` で変更を追えるようにする
4. **修正後に再実行して確認する** — 終了コード `0` になるまで繰り返す
5. **自動修正しない** — スクリプトは検出専用であり、データを書き換えません
